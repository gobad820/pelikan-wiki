> [!abstract] 핵심 요약
> DB 키는 튜플을 식별하기 위한 속성(조합)이다. 유일성·최소성·효율성 보장 여부에 따라 계층 관계를 이룬다.

## 키 계층 구조 (포함 관계)

```
모든 속성 조합
        │
        ▼
  ┌─────────────┐
  │  Super Key  │  유일성만 만족 (최소성 X)
  └──────┬──────┘
         │ 최소성 추가
         ▼
  ┌─────────────────┐
  │  Candidate Key  │  유일성 + 최소성
  └──────┬──────────┘
         │
    ┌────┴────┐
    ▼         ▼
Primary    Alternate
  Key        Key
(선택 1개)  (나머지)

         별개 개념
    ┌────────────┐
    │ Foreign Key│  다른 테이블의 PK 참조
    └────────────┘
```

---

## 각 키 상세 설명

### 1. Super Key (슈퍼 키)

- **조건**: 유일성만 만족
- **최소성**: 불필요한 속성 포함 가능
- **예시**: `학생` 테이블에서
  - `{학번}`, `{학번, 이름}`, `{학번, 이름, 나이}` → 모두 슈퍼 키
  - `{학번}`만 있어도 식별 가능하지만 이름, 나이를 추가해도 여전히 슈퍼 키

```
Super Key ⊇ Candidate Key
```

---

### 2. Candidate Key (후보 키)

- **조건**: 유일성 + 최소성 모두 만족
- **의미**: Primary Key가 될 **자격**이 있는 키
- **예시**: `학생` 테이블에서
  - `{학번}` → 후보 키 ✅
  - `{주민번호}` → 후보 키 ✅
  - `{이메일}` → 후보 키 ✅
  - `{학번, 이름}` → 최소성 위반 ❌ (슈퍼 키에만 해당)

```
Candidate Key ⊂ Super Key
```

> [!tip] 유일성 vs 효율성
> Candidate Key는 유일성·최소성은 만족하지만, **효율성(인덱스)은 보장하지 않는다.**
> Primary Key로 지정되어야 비로소 클러스터드 인덱스가 붙는다.

---

### 3. Primary Key (기본 키)

- **조건**: 후보 키 중 DBA가 **선택한 하나**
- **추가 제약**: `NULL` 불허, 중복 불허
- **효율성**: 클러스터드 인덱스 자동 생성 ✅
- **예시**: 후보 키 `{학번, 주민번호, 이메일}` 중 `{학번}` 선택

```
Primary Key ⊂ Candidate Key
```

> [!info] 유일성 + 효율성 동시 보장
> DB에서 유일성과 효율성을 **항상** 동시에 만족하는 유일한 키.
> Candidate Key는 유일성만, Index는 효율성만 개별 보장.

---

### 4. Alternate Key (대체 키)

- **조건**: 후보 키 중 Primary Key로 **선택되지 못한** 나머지
- **특성**: 후보 키와 동일한 유일성·최소성 보유, 인덱스 자동 생성 ❌

```
Alternate Key = Candidate Key - Primary Key
```

---

### 5. Foreign Key (외래 키)

- **조건**: 다른 테이블의 Primary Key를 **참조**하는 속성
- **목적**: 테이블 간 **참조 무결성** 보장
- **특이점**: 유일성 불필요 → 중복값 허용, NULL 허용 (설정에 따라)
- **예시**:

```sql
-- 수강 테이블
학번(FK) → 학생 테이블의 학번(PK) 참조
과목코드(FK) → 과목 테이블의 과목코드(PK) 참조
```

> [!warning] 계층 구조 밖의 개념
> 다른 키들은 한 테이블 내부의 식별 문제를 다루지만,
> Foreign Key는 **테이블 간 관계(참조 무결성)**를 다루는 별개 개념이다.

---

## 전체 비교표

| 키 종류 | 유일성 | 최소성 | NULL 허용 | 인덱스 자동 생성 | 개수 제한 |
|---------|:------:|:------:|:---------:|:----------------:|:---------:|
| Super Key | ✅ | ❌ | 경우에 따라 | ❌ | 여러 개 |
| Candidate Key | ✅ | ✅ | 경우에 따라 | ❌ | 여러 개 |
| Primary Key | ✅ | ✅ | ❌ | ✅ | **1개만** |
| Alternate Key | ✅ | ✅ | 경우에 따라 | ❌ | 여러 개 |
| Foreign Key | ❌ | ❌ | ✅ | ❌ | 여러 개 |

## 면접 포인트

> [!note] 자주 나오는 질문

**Q. DB에서 유일성과 효율성을 항상 만족하는 것은?**
→ **Primary Key**. Candidate Key는 유일성만, 일반 Index는 효율성만 보장. PK만 둘 다 항상 보장.

**Q. Candidate Key와 Primary Key의 차이는?**
→ Candidate Key는 PK가 될 **자격**을 가진 키들의 집합. 그 중 하나를 선택한 것이 PK.
PK는 NULL 불가 + 클러스터드 인덱스 자동 생성이 추가된다.

**Q. Super Key와 Candidate Key의 차이는?**
→ **최소성** 여부. `{학번, 이름}`은 Super Key지만 `{이름}` 제거 후에도 식별 가능하므로 Candidate Key가 아님.

**Q. Foreign Key의 특징은?**
→ 유일성 불필요(중복 허용), NULL 허용, 테이블 간 참조 무결성 보장.
다른 키들이 한 테이블 내 식별 문제를 다루는 것과 달리, FK는 테이블 간 관계를 정의.
