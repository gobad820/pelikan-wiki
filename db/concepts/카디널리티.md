> [!abstract] 핵심 요약
> 카디널리티는 두 가지 맥락에서 사용된다. ① 테이블 간 관계의 수적 비율 (1:1, 1:N, M:N) ② 컬럼 내 고유값의 수 → 인덱스 효율 결정.

---

## 1. 관계 카디널리티 — 테이블 간 수적 비율

두 엔티티(테이블) 간에 **몇 대 몇 관계**인지를 나타낸다.

| 관계 | 설명 | 예시 |
|------|------|------|
| **1:1** | 한 행이 다른 테이블의 딱 한 행과 대응 | 사람 ↔ 여권 |
| **1:N** | 한 행이 다른 테이블의 여러 행과 대응 | 부서 ↔ 직원 |
| **M:N** | 여러 행이 서로 여러 행과 대응 | 학생 ↔ 수업 |

```
[부서] 1 ──────< N [직원]
부서 하나에 직원 여럿, 직원 하나는 부서 하나에만 소속
```

### M:N은 중간 테이블로 분해

M:N 관계는 RDBMS에서 직접 표현 불가 → 중간(연결) 테이블로 1:N + N:1로 분해

```sql
-- 학생(M) : 수업(N)
학생(학번 PK) ─< 수강(학번 FK, 과목코드 FK) >─ 수업(과목코드 PK)
```

---

## 2. 컬럼 카디널리티 — 고유값의 수

컬럼에 **중복 없이 몇 종류의 값이 있는가**를 나타낸다.

| 카디널리티 | 예시 | 고유값 수 |
|-----------|------|----------|
| **높음 (High)** | 주민번호, 이메일, PK | 거의 모든 행이 다름 |
| **중간 (Medium)** | 이름, 우편번호 | 어느 정도 중복 |
| **낮음 (Low)** | 성별(M/F), 등급(A/B/C), 불리언 | 몇 가지 값만 반복 |

---

## 인덱스와 카디널리티의 관계

카디널리티가 **높을수록** 인덱스 효율이 **좋다**.

```sql
-- 카디널리티 낮음 → 인덱스 거의 무의미
SELECT * FROM user WHERE gender = 'M';
-- 전체의 50%가 해당 → Full Scan이 나을 수도 있음

-- 카디널리티 높음 → 인덱스 매우 효과적
SELECT * FROM user WHERE email = 'test@test.com';
-- 1건만 해당 → 인덱스로 O(log N) 탐색
```

| 카디널리티 | 인덱스 효과 | 이유 |
|-----------|------------|------|
| 높음 | ✅ 매우 좋음 | 인덱스로 소수 행만 접근 |
| 낮음 | ❌ 거의 없음 | 어차피 대부분 행을 읽어야 함 |

> [!tip] 실무 기준
> 카디널리티가 전체 행의 **10~15% 이하**로 좁혀지지 않으면 옵티마이저가 인덱스를 무시하고 Full Table Scan을 선택하기도 한다.

### 복합 인덱스에서의 순서 기준

```sql
-- dept(카디널리티 낮음), email(카디널리티 높음) 복합 인덱스 시
CREATE INDEX idx ON user(email, dept);  -- 카디널리티 높은 컬럼을 앞에 ✅
```

카디널리티가 높은 컬럼을 앞에 배치해야 먼저 많이 걸러내 탐색 범위가 줄어든다.

---

## 면접 포인트

> [!note] 자주 나오는 질문

**Q. 카디널리티가 낮은 컬럼에 인덱스를 걸면 안 되는 이유?**
→ 인덱스로 찾아도 결국 대부분의 행을 읽어야 하므로, 인덱스 탐색 비용만 추가되고 성능 이점이 없다. 옵티마이저가 Full Scan을 선택하기도 한다.

**Q. 복합 인덱스에서 컬럼 순서를 어떻게 정해야 하나?**
→ 카디널리티가 높은 컬럼을 앞에 배치. 먼저 많이 걸러낼수록 탐색 범위가 줄어든다.

**Q. M:N 관계를 DB에서 어떻게 표현하나?**
→ 직접 표현 불가. 중간(연결) 테이블을 두어 1:N + N:1 두 개의 관계로 분해한다.
