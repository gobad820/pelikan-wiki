> [!abstract] 핵심 요약
> 정규화는 데이터 중복을 제거하고 이상(Anomaly) 현상을 방지하기 위해 테이블을 체계적으로 분해하는 과정이다. 1NF → 2NF → 3NF → BCNF 순으로 조건이 강해진다.

---

## 이상(Anomaly) 현상 — 정규화가 필요한 이유

| 이상 종류 | 설명 | 예시 |
|-----------|------|------|
| **삽입 이상** | 불필요한 데이터 없이는 삽입 불가 | 수강생 없는 강좌 등록 불가 |
| **삭제 이상** | 하나 삭제 시 의도치 않은 데이터 소실 | 마지막 수강생 삭제 시 강좌 정보도 사라짐 |
| **갱신 이상** | 중복 데이터 일부만 수정 → 불일치 | 교수 연락처가 여러 행에 있는데 한 곳만 수정 |

---

## 제1정규형 (1NF)

**조건**: 모든 속성의 도메인이 **원자값(Atomic Value)**이어야 한다.

```
❌ 비정규형
┌─────┬──────────────────┐
│학번 │수강과목           │
├─────┼──────────────────┤
│S001 │수학, 영어, 물리   │  ← 다중값 (원자값 위반)
└─────┴──────────────────┘

✅ 1NF 변환
┌─────┬────────┐
│학번 │수강과목 │
├─────┼────────┤
│S001 │수학     │
│S001 │영어     │
│S001 │물리     │
└─────┴────────┘
```

---

## 제2정규형 (2NF)

**조건**: 1NF를 만족하고, **부분 함수 종속을 제거**해야 한다.

- 부분 함수 종속: 복합 PK 중 **일부**에만 종속되는 속성이 있는 경우

```
❌ 1NF (부분 종속 존재)
수강(학번, 과목코드, 성적, 교수명)
         PK ─────────────────────┘
    └────────┘             │
     복합 PK       교수명은 과목코드에만 종속 (부분 종속)

✅ 2NF 변환 (테이블 분리)
수강(학번, 과목코드, 성적)   ← 완전 함수 종속만 남김
과목(과목코드, 교수명)       ← 부분 종속 분리
```

> [!tip] 2NF는 복합 PK가 있을 때만 의미있다
> PK가 단일 컬럼이면 부분 종속이 발생할 수 없으므로 자동으로 2NF 만족.

---

## 제3정규형 (3NF)

**조건**: 2NF를 만족하고, **이행 함수 종속을 제거**해야 한다.

- 이행 함수 종속: A → B → C 관계에서 A → C가 성립하는 경우

```
❌ 2NF (이행 종속 존재)
학생(학번, 학과코드, 학과명)
학번 → 학과코드 → 학과명  (이행 종속: 학번 → 학과명)

✅ 3NF 변환
학생(학번, 학과코드)
학과(학과코드, 학과명)
```

---

## BCNF (보이스-코드 정규형)

**조건**: 3NF를 만족하고, **모든 결정자가 후보 키**여야 한다.

3NF보다 엄격한 버전. 후보 키가 아닌 속성이 결정자가 되면 위반.

```
❌ 3NF지만 BCNF 위반
수강(학생, 과목, 교수)
  규칙: 한 과목은 한 교수만 담당, 학생은 과목당 한 교수에게만 수강
  후보 키: (학생, 과목) 또는 (학생, 교수)
  교수 → 과목 (교수가 결정자인데 후보 키가 아님) ← BCNF 위반

✅ BCNF 변환
수강(학생, 교수)
교수담당(교수, 과목)
```

## 정규형 단계 요약

```
비정규형
   │  원자값 보장
   ▼
  1NF
   │  부분 함수 종속 제거 (복합 PK의 일부에만 종속)
   ▼
  2NF
   │  이행 함수 종속 제거 (A→B→C)
   ▼
  3NF
   │  모든 결정자 = 후보 키
   ▼
 BCNF    ← 실무에서는 주로 여기까지
   │
  4NF (다치 종속 제거)
   │
  5NF (조인 종속 제거)
```

| 정규형 | 핵심 조건 | 제거 대상 |
|--------|-----------|-----------|
| **1NF** | 원자값 | 다중값 속성, 반복 그룹 |
| **2NF** | 1NF + 완전 함수 종속 | 부분 함수 종속 |
| **3NF** | 2NF + 직접 종속 | 이행 함수 종속 |
| **BCNF** | 모든 결정자 = 후보 키 | 후보 키 아닌 결정자 |

## 역정규화 (Denormalization)

정규화를 일부러 되돌리는 것. 정규화가 무조건 좋은 것은 아니다.

| 항목 | 정규화 | 역정규화 |
|------|--------|----------|
| 데이터 중복 | 적음 | 많음 |
| 갱신 성능 | 좋음 | 나쁨 (중복 갱신 필요) |
| 조회 성능 | 나쁨 (JOIN 증가) | 좋음 |
| 사용 상황 | OLTP (트랜잭션 많음) | OLAP (분석·집계 많음) |

> [!warning] 역정규화 시 주의
> 역정규화는 의도적 선택이어야 한다. 무분별하게 적용하면 갱신 이상이 다시 발생한다.

## 면접 포인트

> [!note] 자주 나오는 질문

**Q. 정규화를 왜 하는가?**
→ 데이터 중복 제거 + 삽입/삭제/갱신 이상 현상 방지. 데이터 무결성 확보.

**Q. 3NF와 BCNF의 차이?**
→ 3NF는 "비주요 속성이 후보 키에 이행 종속되면 안 된다"이고, BCNF는 더 엄격하게 "모든 결정자가 반드시 후보 키여야 한다". BCNF ⊂ 3NF.

**Q. 실무에서 항상 BCNF까지 정규화해야 하는가?**
→ 아니다. 조회 성능이 중요한 경우(OLAP, 리포트) 역정규화를 의도적으로 적용한다. 정규화와 성능은 트레이드오프 관계.

**Q. 2NF가 의미있으려면?**
→ 복합 PK가 있어야 한다. PK가 단일 컬럼이면 부분 종속이 원천적으로 불가능해 2NF를 자동 만족.
