## 오늘 학습한 개념

- [[FrontControllerPattern]] — 단일 서블릿이 action 파라미터로 요청을 분기 처리
- [[Filter]] — 서블릿 전/후에 공통 처리 (로깅, XSS 필터링), Filter Chain 패턴
- [[Listener]] — 웹앱 이벤트 모니터링, `ServletContextListener`로 공유 자원 초기화

## 핵심 코드 스니펫

### Front Controller — switch와 action 파라미터

```java
@WebServlet("/main")
public class MainServlet extends HttpServlet implements ControllerHelper {

    protected void doGet(HttpServletRequest request, HttpServletResponse response)
            throws ServletException, IOException {
        String action = preProcessing(request, response);
        switch (action) {
        case "guguform":
            guguform(request, response);
            break;
        case "gugu":
            gugu(request, response);
            break;
        }
    }
}
```

### XSS Filter — HttpServletRequestWrapper

```java
class XssRequestWrapper extends HttpServletRequestWrapper {

    private String sanitize(String value) {
        if (value == null) return null;
        return value.replace("<", "&lt;").replace(">", "&gt;");
    }

    @Override
    public String getParameter(String name) {
        return sanitize(super.getParameter(name));
    }

    @Override
    public Map<String, String[]> getParameterMap() {
        Map<String, String[]> filtered = new HashMap<>();
        super.getParameterMap().forEach((k, v) ->
            filtered.put(k, Arrays.stream(v).map(this::sanitize).toArray(String[]::new)));
        return Collections.unmodifiableMap(filtered);
    }
}
```

### Listener — 웹앱 시작/종료 훅

```java
@WebListener
public class MyContextListener implements ServletContextListener {
    public void contextInitialized(ServletContextEvent sce) {
        // 서버 시작 시 1회: DB ConnectionPool 등 무거운 자원 초기화
    }
    public void contextDestroyed(ServletContextEvent sce) {
        // 서버 종료 시 1회: 자원 반납
    }
}
```

## 핵심 개념 정리

### Servlet 3대 컴포넌트

| 컴포넌트 | 어노테이션 | 역할 |
|---------|-----------|------|
| Servlet | `@WebServlet` | 요청 처리, 응답 생성 |
| Filter | `@WebFilter` | 요청/응답 전/후처리 (체인 구조) |
| Listener | `@WebListener` | 이벤트 모니터링 (시작/종료/세션 등) |

### Filter Chain 흐름

```
Client → [Filter1: Logging] → [Filter2: XSS] → [Servlet] → [Filter2] → [Filter1] → Client
         chain.doFilter()      chain.doFilter()              ← 응답 방향
```

### Switch 화살표 문법과 Checked Exception (Java 21 기준)

| 구분 | 화살표 switch 문(statement) | 화살표 switch 식(expression) |
|------|--------------------------|---------------------------|
| Checked Exception 전파 | **가능** | 불가 |
| Unchecked Exception | 가능 | 가능 |

- **Switch 문(statement)**: `switch(x) { case "a" -> foo(); }` — 값 반환 없음, Checked Exception 전파 가능
- **Switch 식(expression)**: `var r = switch(x) { case "a" -> 1; }` — 값 반환, Checked Exception 전파 불가 (람다 맥락)

> Java 14+에서 switch STATEMENT의 화살표 문법은 람다가 아니므로 `throws` 선언된 Checked Exception이 정상 전파된다.

### Filter Chain 내부 동작 (ApplicationFilterChain)

Tomcat은 `ApplicationFilterChain` 클래스로 필터 체인을 구현한다.

```
chain.doFilter() 호출
    → ApplicationFilterChain.doFilter() 실행
    → 남은 필터 인덱스 확인
    → 필터 남아 있으면: 다음 Filter.doFilter() 호출
    → 필터 소진되면: servlet.service(request, response) 직접 호출
```

### HttpServlet 템플릿 메서드 패턴

```
컨테이너 → HttpServlet.service() [상속, 오버라이드 안 함]
                ↓ HTTP Method 판별
           MyServlet.doGet() [오버라이드된 내 메서드]
```

`service()`는 오버라이드하지 않고, `doGet()`/`doPost()`만 오버라이드하면 되는 이유가 **템플릿 메서드 패턴** — `HttpServlet.service()`가 분기 처리를 담당한다.

### @Override 어노테이션의 역할

| 구분 | 설명 |
|------|------|
| 역할 | 컴파일러에게 "부모 메서드를 오버라이드한다"고 알리는 검증용 어노테이션 |
| 없어도 동작? | **동작함** — 오버라이드 자체는 메서드 시그니처가 일치하면 자동 적용 |
| 없을 때 위험 | 시그니처 오타 시 컴파일러가 잡지 못해 새 메서드로 인식될 수 있음 |

### HttpFilter.doFilter(HttpServletRequest...) 자동 생성 안 되는 이유

Eclipse "Add unimplemented methods"는 **인터페이스 추상 메서드**만 자동 생성한다.
`HttpFilter`의 `doFilter(HttpServletRequest, HttpServletResponse, FilterChain)`은 protected **구체 메서드**이므로 자동 생성 대상이 아니다.
Source → **Override/Implement Methods** 메뉴에서 직접 선택해야 한다.

### getParameterMap() 출력 시 해시값 문제

```java
// 잘못된 방식 — Map<String, String[]>을 toString()하면 해시값 출력
System.out.println(request.getParameterMap());

// 올바른 방식 — forEach + Arrays.toString으로 배열 값 출력
request.getParameterMap().forEach((k, v) ->
    System.out.printf("[LOG] %s = %s%n", k, Arrays.toString(v)));
```

`String[]`은 `toString()` 시 배열 내용이 아닌 해시값(`[Ljava.lang.String;@...`)이 출력된다.
`Arrays.toString(v)`를 사용해야 `[value1, value2]` 형태로 출력된다.

## 트러블슈팅 기록

| 문제 | 원인 | 해결 |
|------|------|------|
| `java.io.UnsupportedEncodingException: uft-8` | `setContentType("text/html;charset=uft-8")` 오타 | `utf-8`로 수정 |
| XSS Filter 로그에 미필터링 값 출력 | `getParameterMap()` 미오버라이드 | `getParameter`, `getParameterValues`, `getParameterMap` 모두 오버라이드 |
| STS 자동 재로드 미동작 | JVM Hot Swap 한계 + 메서드 시그니처/어노테이션 변경 | 서버 재시작 필요. 단순 메서드 내부 변경만 자동 반영 |
| `404 /main` 리소스 없음 | 소스 수정 후 Tomcat에 미배포 | Eclipse Servers 탭 → Publish |
| `getParameterMap()` 출력 시 해시값 | `String[]`의 `toString()`은 배열 내용 미출력 | `Arrays.toString(v)` 사용 |
| FrontController `hello`/`sum` 동작 혼용 | 메서드명과 구현 내용이 뒤바뀜 (sum→hello, hello→sum) | 메서드명 또는 내부 구현 교정 |
| `404 /front-controller-app/front` | 프로젝트가 Tomcat에 미배포 상태 | Servers 탭 → Add and Remove → Publish |

## 오늘의 인사이트

> [!quote] 핵심 기억
> Filter는 Servlet의 앞/뒤를 감싸는 `데코레이터 패턴`.
> `HttpServletRequestWrapper`로 요청을 래핑하면 파라미터를 투명하게 가공할 수 있다.
> 단, `getParameter()` 하나만 오버라이드하면 `getParameterMap()`으로 접근 시 원본이 노출된다 — 세 메서드 모두 오버라이드해야 완전한 필터링이 적용된다.
>
> Java 14+에서 switch **문(statement)** 의 화살표 문법은 람다가 아니므로 Checked Exception이 정상 전파된다.
> Checked Exception 전파 불가는 **switch 식(expression)** 에서만 해당된다 — 값을 반환하는 맥락에서만 제약이 있다.
>
> `chain.doFilter()`는 Tomcat의 `ApplicationFilterChain`이 관리한다. 필터를 모두 소진하면 컨테이너가 직접 `servlet.service()`를 호출하고, 개발자가 오버라이드한 `doGet()`/`doPost()`로 위임된다.
